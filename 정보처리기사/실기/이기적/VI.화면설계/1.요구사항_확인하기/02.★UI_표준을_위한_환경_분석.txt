출제빈도 중
빈출 태그 : UI 요구사항 확인 절차 / UI 요구사항 요소

1) 사용자 경향 분석
- 기존~현존 UI 경향을 숙지하고 현재 UI의 단점을 작성함
- 사용자의 요구사항을 파악하고, 쉽게 이해 가능한 기능 위주로 기술 영역을 정의함

2) 기능 및 설계 분석
- 기능 조작성 분석
    - 사용자 편의를 위한 조작에 대해 분석함
    - ex) 스크롤 바 지원 가능 여부, 마우스 조작 시 동선 확인
- 오류 방지 분석
    - 조작 시 오류에 대해 예상 가능한지 확인
    - ex) 의도치 않은 페이지 이동 여부, 기능 버튼의 명확한 구분이 가능한지 확인, 기능 버튼의
        이름이 사용자 조작과 일치하는지 확인
- 최소한의 조작으로 업무 처리가 가능한지 형태 분석
    - 작업 흐름에 가장 적합한 레이아웃인지 확인
    - ex) 기능 특성에 맞는 UI 확인 및 조작 단계 최소화와 동선 단순 여부 확인
- UI의 정보 전달력 확인
    - 중요 정보의 인지가 쉽게 전달 가능한지, 정보 제공의 간결성과 명확성을 확인하고 정보 제공
        방식의 일관성과 사용자 이해성 확인, 상호연관성이 높은 정보인지 확인
    - ex) 오류 발생 시 해결 방법 접근 용이성 확인

3) UI 요구사항 확인 절차
- 요구사항 확인 절차
    - 비즈니스 요구사항 확인
        (1) 목표 정의
            - 실사용자 대상 심층 인터뷰를 통해 의견 수렵하고, 비즈니스 요구사항을 정의함
            - 인터뷰를 통해 사업적, 기술적 요소를 깊게 이해해 목표를 명확히 한 뒤 사업적,
                기술적 목표가 확정되면 UI/UX 디자인 프로세스를 정의함
            - 요구사항 인터뷰
                - 사업적 요구사항 파악
                - 기술적 요구사항 파악
            - 인터뷰 유의사항
                - 개별적 진행
                - 가능한 다수의 의견수렴
                - 리서치 시작 전 시행
                - 1시간을 넘기지 않음
            - 비즈니스 요구사항 정의
        (2) 활동사항 정의
            - 목표 정의에서 조사한 요구사항을 바탕으로 진행해야 할 활동을 정의함
            - 사용자, 고객, 회사의 비전을 일치시키는 작업을 통해 초기 비전과 기대를 정의함
            - UX 조사와 UI/GUI 디자인에 필요한 예산과 일정을 결정해 비용과 일정을 확인함
            - 기술 발전 가능성을 파악하고 UI/UX 디자인의 나아갈 방향을 제시함
            - 사업 전략/사업 목표, 프로세스 책임자 선정, 회의 일정 및 계획 작성, 우선순위 선정,
                개별 단위 업무를 구분함
        (3) 인터뷰 진행
            - 인터뷰의 목적 : 인터뷰 내용을 토대로 경영진 내의 서로 다른 UI/UX 개발 프로젝트의
                활동 사항을 설명하고 협의하도록 도움
            - 가능하다면 단체보다는 개별로 진행하는 것이 좋음
            - 다수의 의견에 집중하면 개인의 중요한 의견을 놓칠 수 있으므로 주의함
            - 될 수 있는 한 많은 인원을 인터뷰함
            - 인터뷰 시간은 한 시간이 넘지 않도록 함
    - 요구사항 작성
        (4) 요구사항 요소 확인
            - 실사용자를 대상으로 다양한 방법으로 수집하고 작성된 요구사항을 검토함
            - 목적을 기준으로 요구사항을 작성하고, 요구사항을 바탕을 UI 전체 구조를 파악해야 함
            - 요구사항 작성 순서 : 요구사항 요소 확인 -> 정황 시나리오 작성 -> 요구사항 작성
            - 요구사항 요소
                - 데이터 요구
                    - 사용자 요구 모델과 객체들의 핵심 특성에 기반해 데이터 객체를 정리함
                    - 인터페이스에 영향을 줄 수 있으니 초기에 확인함
                    - ex) Email 메시지 속성 : 제목, 송신자, 송신일, 참조인, 답변 등
                - 기능 요구
                    - 동사형으로 사용자의 목적 달성을 위해 실행해야 할 기능을 설명함
                    - 기능 요구 목록으로 정리함
                    - 최대한 철저하게 작성해야 함
                    - ex) 사용자는 메일을 작성하거나, 수신하고, 참조해 발송할 수 있음
                - 제품/서비스 품질
                    - 감성 품질과 데이터/기능 요구 외 제품 품질, 서비스 품질을 고려함
                    - 시스템 처리 능력 등 정량화 가능한 요구사항을 확인함
                - 제약 사항
                    - 비용, 데드라인, 시스템 준수에 필요한 규제 등을 확인함
                    - 사전에 제약 사항의 변경 여부를 확인함
        (5) 정황 시나리오 작성
            - 개발하는 서비스의 초기 모양을 상상하는 단계
            - 사용자 관점에서 작성하며 요구사항 정의에서 가장 기초적인 시나리오를 의미함
            - 높은 수준과 낙관적인 상황에서 이상적 시스템 동작에 초점을 둠
            - 육하원칙을 따르고 사용자가 주로 사용하는 기능을 기반으로 작성함
            - 간단명료하게 작성해 정확하게 전달함
            - 같이 동작하는 기능은 하나의 시나리오에 통합함
            - 외부 전문가, 경험이 많은 사람에게 검토를 의뢰하도록 함

4) UI 시나리오 문서의 기대 효과
- 요구사항이나 의사소통에 대한 오류를 감소시킴
- 개발 과정 중의 혼선을 최소화해 개발 속도를 향상시킴
- 불필요한 기능을 최소화해 시나리오 작성과 소프트웨어 개발 비용을 줄일 수 있음
- 유관 부서 만족도를 높임
