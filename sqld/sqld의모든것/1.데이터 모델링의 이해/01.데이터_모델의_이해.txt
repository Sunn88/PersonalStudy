1. 모델링의 개념
- 모델링 : 현실세계를 대상으로 일종의 모델을 만드는 것
    - 모델 : 현실세계의 사물 혹은 개념을 구성요소로 나누고 이를 일정하게 도식화한 것
    - 도식화하는 방법이 약속된 표기법을 따라야만 함
    - 업무의 흐름을 가시화하고 명세화해 이를 설계, 개발 또는 시스템 관리에 사용
- 모델링은 추상화를 기본으로 함
    - 추상화 : 사물의 특징을 추려서 단순화해 나타내는 것
        - 어떤 특징을 어느 정도로 표현할지는 모델링의 목적에 따라 달라짐
        - 필요한 수준에서 데이터를 추상화해 단순명료하게 표현하는 것이 중요
- 데이터베이스 모델링 : 정보시스템을 구축하기 위해 데이터베이스를 설계하는 것도 하나의 모델링

2. 모델링의 특징
- 추상화(Abstraction) : 대상의 주요 특징을 추출해 일정한 방식으로 표현함
- 단순화(Simplification) : 복잡한 현실세계를 보다 단순하게 표현함 -> UML, ERD같이 제한된 언어나 약속된 표기법 사용
- 명확화(Clarity) : 보는 사람에 따라 서로 다르게 해석되지 않고 대상을 명확하게 이해할 수 있도록 모호함이 없게 기술함

3. 모델링의 세 가지 관점
- 데이터 관점(대상, What)
    - 업무를 구성하는 데이터에 집중
    - 어떤 데이터들이 서로 관계를 맺고 사용되는지를 모델링하는 것
    - 정적분석, 구조분석 등을 기반
- 프로세스 관점(처리방법, How)
    - 업무의 흐름에 집중
    - 업무가 실제로 처리하는 일이 어떻게 처리되는지 모델링
    - 동적분석, 도메인분석 등을 기반
- 데이터와 프로세스의 상관 관점(대상과 처리방법의 상관관계, Interaction)
    - 업무를 구성하는 데이터와 프로세스가 서로 어떻게 관계를 맺고 영향을 주고받는지를 모델링
    - CRUD 분석을 기반

4. 모델링의 세 가지 단계
- 개념적 데이터 모델링(Conceptual Data Modeling)
    - 가장 높은 추상화 레벨을 가진 모델링
    - 업무와 개념 중심으로 포괄적인 수준에서 모델링을 수행함
    - 전사적 차원의 데이터 모델링을 수행할 때 이루어짐
    - EA(Enterprise Architecture)를 수립할 때 많이 이용됨 -> 이 단계에서 엔티티(Entity)와 속성(Attribute)을 도출함
- 논리적 데이터 모델링(Logical Data Modeling)
    - 데이터 모델에 대한 키(Key), 속성(Attribute), 관계(Relationship) 등을 표현
    - 다른 DBMS에 적용이 가능한 수준에서의 추상화 레벨을 가짐 -> 재사용성이 높음
    - 정규화를 통해 중복 데이터를 최소화함 -> 이 단계에서 식별자를 도출하고 관계를 정의함
- 물리적 데이터 모델링(Physical Data Modeling)
    - 특정 DBMS에 맞추어 구현이 가능한 수준에서 모델링을 수행함
    - DBMS의 성능이나 보안, 가용성 등을고려해 설계
    - 가장 낮은 수준에서의 추상화 레벨을 가짐
    - 성능 향상을 위해 반정규화를 수행 -> 테이블, 인덱스, 함수 등을 생성
    
5. ANSI-SPARC에서 정의한 3단계 스키마 구조
- 외부 스키마(External Schema)
    - 사용자 관점
    - 사용자 또는 애플리케이션이 바라보는 데이터베이스 스키마 정의
    - 다중 사용자 뷰 제공
- 개념 스키마(Conceptual Schema)
    - 설계자 관점
    - 모든 사용자가 바라보는 데이터베이스 스키마를 통합해 나타냄
    - 전체 데이터베이스에 저장되는 데이터와 그 관계 정의
    - 통합된 뷰 제공
- 내부 스키마(Internal Schema)
    - 개발자 관점
    - 디스크나 메모리 상의 물리적, 실질적 저장구조를 나타냄
    - 테이블, 칼럼, 인덱스 등 정의
    - 물리적 뷰 제공
- 3단계 스키마 구조를 통해 보장되는 데이터 독립성
    - 논리적 데이터 독립성 : 외부 스키마와 개념 스키마 간의 독립성 -> 개념 스키마가 변경되어도 외부 스키마는 영향을 받지 않음
    - 물리적 데이터 독립성 : 개념 스키마와 내부 스키마 간의 독립성 -> 내부 스키마가 변경되어도 개념 스키마나 외부 스키마는 영향을 받지 않음

6. ERD
- ERD : 데이터베이스를 구성하는 데이터의 논리적 구성요소를 엔티티로 정의하고, 엔티티들 간의 관계를 특별한 표기법으로 나타냄으로써 스키마를 설계하는 방법 제공
- ERD 작성순서
    - 엔티티 도출
    - 도출된 엔티티를 적절히 배치, 중요한 엔티티는 왼쪽 상단 배치
    - 엔티티 간의 관계 설정
    - 관계명 기술(행위 관계, 존재 관계 표현)
    - 관계의 참여도(Cardinality) 기술(일대일, 일대다, 다대다 표현)
    - 관계의 필수/선택 여부 기술(Null 값 가질 수 있는지 여부)
